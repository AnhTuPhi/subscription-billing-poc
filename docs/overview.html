Subscription Billing POCs

Three runnable Spring Boot 3.4 / Java 21 POCs, each isolating one genuinely hard part of SaaS recurring billing. This page is a visual walkthrough of what each one protects, how it works, and what changes when it runs behind many pods. See ISSUE.md, TECHNICAL.md, and CONSISTENCY.md for the prose.

What we are protecting

Every path here moves money. A bug doesn't misplace a pixel — it overcharges a customer, drops revenue, or issues a refund that should never have left. Five invariants hold across all three POCs; everything in the code exists to protect one of them.

I1 · AuditableEvery amount is reproducible from its inputs and each line item explains itself.
I2 · No double-chargeA retried renewal or replayed usage event never bills twice.
I3 · No dropped revenueA late or duplicated event is still counted exactly once; a recoverable failure is retried.
I4 · No wrong money outA downgrade produces credit, never a cash refund.
I5 · Legal & idempotent stateSubscriptions move only along allowed edges; re-running a batch changes nothing the second time.

The recurring pattern

One idea ties the three together and makes the scaling story tractable:

The database enforces the invariants that must survive concurrency; the application enforces the policy. DB-unique idempotency, rebuildable + reconciled aggregates, and an explicit state machine are the load-bearing choices.

2 · Dunning localhost:8082

Hard problem: a renewal charge is declined. Don't cancel on the first failure (you'd drop recoverable revenue), don't retry forever (fees, resentment). Move through a lifecycle with a deterministic retry cadence, notify at each stage, let the customer fix their card and settle instantly — and make the clock safe to re-run.

The state machine

ACTIVE →charge fails PAST_DUE →grace elapsed SUSPENDED →window elapsed CANCELED
↺ from PAST_DUE / SUSPENDED ACTIVE card updated → charge succeeds → pending retries canceled

Retry cadence application.yml

retry-offsets-days[3, 5, 7] — attempt #2 at day 3, #3 at day 5, #4 at day 7 after the failure
grace-period-days2 — PAST_DUE → SUSPENDED after retries exhausted + grace
suspension-window-days14 — SUSPENDED → CANCELED

Tech debt to acknowledge

  • tick is triggered by an API call, not a scheduler — needs exactly-once execution across pods.
  • No idempotency key sent to the charge gateway; findDueRetries needs FOR UPDATE SKIP LOCKED for multi-pod safety.
  • Time is a request parameter (today), not a timezone-aware clock.
  • Notifications are synchronous + in-memory — no outbox, so a crash can lose (or a retry duplicate) an email.

Why each design choice

Idempotent ingestevent_id UNIQUE + catch DataIntegrityViolationException — the DB is the arbiter, safe across N writers as-is
Rebuildable rollupdelete + recompute per (customer, metric, day) → late/out-of-order events are a non-event
Deterministic bucketusageDay derived from occurredAt in UTC
Transparencyeach TierCharge → line item carrying qty + price → customer re-derives the amount
Proof of honestyReconciliationService compares raw vs rollup per metric

Tech debt to acknowledge

  • Rollup delete+insert isn't atomic vs concurrent ingest — reconciliation is the deliberate safety net.
  • Rollup is manually triggered; needs scheduled/streaming rollup at scale.
  • Full-range scans, no partitioning — real "millions" needs time-series/columnar storage.
  • Reconciliation detects drift but does not self-heal; pricing is static + global.

Scaling to N pods / a VM fleet

Today: one process, in-memory H2. That single-node assumption hides every distributed problem. Here's what breaks and what stays correct behind replicas: N.

Step zero: h2:mem is per-process — two pods = two truths. Nothing works until every replica shares one PostgreSQL/Oracle.

EASY Stateless request tier

Controllers/services hold no in-process state and no HTTP session. Any pod serves any request. Create-sub, change-plan, ingest, queries — scale horizontally freely, no sticky sessions.

HARD Batch / clock tier

dunning.tick, rollup, and end-of-cycle renewals sweep the dataset on a schedule. Naive @Scheduled on every pod → N concurrent runs → double-charge. Must run exactly once.

Exactly-once batch — pick one

A · Externalize start herek8s CronJob (or a replicas:1 worker) calls the batch. One firing, one runner. Web tier stays stateless.
B · Leader electionShedLock / clustered Quartz / k8s Lease — only the leader acts.
C · PartitionShard by customerId % N; each pod owns a disjoint slice.

Consistency levels we promise

Subscription / invoice / creditStrong ACID + row/optimistic locks
Raw usage eventsExactly-once storage unique constraint
Rollups vs raw eventsEventually consistent rebuildable + reconciled on schedule
Notifications / chargeAt-least-once / effectively-once outbox + idempotency key

Financial state is strongly consistent. Derived data is eventually consistent by design and continuously reconciled — drift is measured, not assumed absent. Full checklist in CONSISTENCY.md.
